# Технический SEO и Schema.org

Дата проверки: 22.07.2026  
Контур: `http://127.0.0.1:3001`, исходники Next.js 15.5.21  
Ограничение: заголовки проверялись на локальном dev-сервере; HTTPS/HSTS и CDN-кеширование следует повторно проверить на боевом домене.

## Находки по приоритету

### High — canonical отсутствует на всех индексируемых страницах

**Доказательство:** в фактическом HTML `/`, `/privacy`, `/consent` найдено `0` `<link rel="canonical">`; `app/layout.tsx` задаёт `metadataBase`, но не `alternates.canonical`. URL с параметрами (`/?utm_source=test`, `/privacy?x=1`) возвращают `200` и также не содержат canonical.

**Риск:** сигналы могут дробиться между URL с UTM/параметрами и вариантами адреса; после подключения собственного домена поисковик может дольше выбирать основной host.

**Исправление:** задать self-referencing canonical через Next Metadata API: `/`, `/privacy`, `/consent`. На продакшене настроить один основной HTTPS-host и постоянные редиректы со всех альтернативных host/протоколов.

### High — временный домен зашит сразу в четырёх SEO-источниках

**Доказательство:** `https://barber-i-barbi.vercel.app` указан в `app/layout.tsx:7`, `app/page.tsx:53`, `public/robots.txt:5` и во всех `<loc>` `public/sitemap.xml:3-5`.

**Риск:** при публикации на другом домене metadata, sitemap и JSON-LD будут сообщать поисковикам противоречивый URL.

**Исправление:** вынести production origin в одну проверяемую переменную/константу и генерировать metadata, robots и sitemap через Next Metadata Routes. До релиза убедиться, что в HTML, JSON-LD, sitemap и robots присутствует один и тот же HTTPS-origin.

### High — незаполненные юридические шаблоны разрешены к индексации и находятся в sitemap

**Доказательство:** `/privacy` и `/consent` возвращают `200`, наследуют `<meta name="robots" content="index, follow">` и включены в sitemap. В исходниках присутствуют явные placeholders: `[ФИО ...]`, `[добавить]`, а также предупреждение «до публикации» (`app/privacy/page.tsx:3`, `app/consent/page.tsx:3`).

**Риск:** в индекс попадут незавершённые страницы с низким доверием и шаблонными данными.

**Исправление:** лучший вариант — заполнить и юридически проверить документы до публикации. Пока они не готовы, поставить для них `noindex, follow` и убрать из sitemap; не закрывать их только в robots.txt, иначе бот не увидит `noindex`.

### Medium — metadata юридических страниц частично дублирует главную

**Доказательство:** title уникальны, но обе страницы наследуют коммерческий description главной («стрижки, окрашивание…»), а также одинаковые OG/Twitter title и description. На главной нет `og:url`, `og:image`, `og:site_name`; Twitter автоматически получает только `summary` без изображения. Title главной — 58 символов, description — 103 символа: размеры приемлемы.

**Исправление:** задать каждой странице собственные description, canonical и OG URL; добавить качественное абсолютное изображение 1200×630, `openGraph.siteName` и Twitter `summary_large_image`. Юридические страницы можно не оптимизировать под соцсети, если они временно `noindex`.

### Medium — Schema.org требует укрепления идентификаторов и локальных данных

**Что работает:** в первоначальном SSR HTML есть синтаксически корректный JSON-LD с `@context`, типом `BeautySalon`, названием, телефоном, адресом, графиком, изображением и `AggregateRating`. Разметка не зависит от выполнения JavaScript. График согласован с видимым контентом: четверг исключён.

**Проблемы и доказательства:**

- `@id` равен URL карточки Яндекс.Карт (`app/page.tsx:53`), хотя устойчивый идентификатор собственной сущности должен принадлежать сайту.
- Яндекс.Карты не указаны как `sameAs` или `hasMap`.
- Нет `geo`, `priceRange`, `logo`, `areaServed`, `WebSite`/`WebPage` и связи сущностей через `@graph`.
- `aggregateRating.reviewCount = 33`, но на странице видны значение `4,8` и только три отзыва; число 33 визуально не показано. Оно должно быть актуальным и доступным пользователю. Кроме того, звёздный rich result для self-serving `LocalBusiness` не гарантируется, поэтому разметку нельзя считать источником сниппета.
- `url` и `image` жёстко привязаны к временному Vercel-host.

**Исправление:** использовать `@graph` с `WebSite`, `WebPage` и `BeautySalon`; назначить бизнесу `@id: https://<основной-домен>/#business`, а Яндекс.Карты перенести в `sameAs`/`hasMap`. Добавить только подтверждённые `geo`, `priceRange`, `logo`, `areaServed`. Показывать на странице тот же `reviewCount`, что в JSON-LD, регулярно обновлять его либо удалить `aggregateRating`, если поддерживать достоверность нельзя. После боевого деплоя проверить Rich Results Test и Schema Markup Validator.

### Medium — защитные HTTP-заголовки явно не настроены

**Доказательство:** локальный ответ содержит `X-Powered-By: Next.js`, но не содержит `Content-Security-Policy`, `X-Content-Type-Options`, `Referrer-Policy`, `X-Frame-Options`/`frame-ancestors` и `Permissions-Policy`. В `next.config.mjs` нет `headers()` и не отключён `poweredByHeader`. `robots.txt` имеет корректный `text/plain`, sitemap — `application/xml`.

**Исправление:** в production добавить CSP (предпочтительно `frame-ancestors` вместо устаревающего X-Frame-Options), `X-Content-Type-Options: nosniff`, строгий `Referrer-Policy`, ограниченный `Permissions-Policy`; отключить `poweredByHeader`. HSTS включать и оценивать только на окончательном HTTPS-домене. Повторить проверку фактических Vercel-заголовков после деплоя. `Cache-Control: no-store` текущего dev-сервера не использовать для оценки production-кеширования.

### Low — 404 корректен, но содержит конфликтующие метатеги

**Доказательство:** неизвестный маршрут возвращает `404` и содержит `noindex`, что правильно. Одновременно в HTML присутствуют два title и два robots-набора: `noindex` плюс унаследованный `index, follow`; наиболее строгая директива должна победить, но сигнал избыточно противоречив.

**Исправление:** добавить собственный `app/not-found.tsx` и его metadata, исключив наследуемый `index, follow` для 404.

### Low — внутренняя перелинковка исправна, но минимальна

**Доказательство:** главная ссылается на `/privacy` и `/consent`, обе страницы возвращают `200` и имеют ссылку назад на `/`; иных внутренних HTML-URL нет. Навигация по секциям использует рабочие якоря. Важные страницы находятся на глубине одного клика. Битых внутренних ссылок в проверенном наборе не найдено. Внешние ссылки на Яндекс.Карты используют `rel="noreferrer"`; телефон размечен `tel:`.

**Исправление:** если появятся отдельные посадочные страницы услуг, связать их из карточек услуг и добавить хлебные крошки плюс `BreadcrumbList`. Для текущей трёхстраничной структуры это не блокер.

### Info — robots и sitemap технически исправны

**Доказательство:** `/robots.txt` отвечает `200 text/plain`, содержит `User-agent: *`, `Allow: /`, `Disallow: /api/` и абсолютную ссылку на sitemap. `/sitemap.xml` отвечает `200 application/xml`, XML парсится без ошибок, все три перечисленных URL соответствуют существующим `200` маршрутам. Специальных запретов для AI-crawlers нет, то есть они наследуют общий доступ.

**Улучшение:** после выбора домена генерировать оба файла динамически; добавить правдивый `lastModified` только при наличии реальной даты изменения. Не добавлять в sitemap `noindex`, редиректы, API и незавершённые документы.

## Рекомендуемый порядок работ

1. Зафиксировать основной production-домен; синхронизировать его в metadata, JSON-LD, robots и sitemap.
2. Добавить canonical на все индексируемые страницы и 301/308-редиректы вариантов host/protocol.
3. Заполнить юридические страницы либо временно перевести их в `noindex, follow` и убрать из sitemap.
4. Сделать уникальные metadata и OG image; очистить metadata 404.
5. Пересобрать JSON-LD вокруг собственного `@id`, проверить рейтинг и добавить подтверждённые локальные свойства.
6. Настроить защитные заголовки и повторно снять заголовки/валидаторы на боевом HTTPS-домене.

## Проверенная поверхность

- URL: `/`, `/privacy`, `/consent`, `/robots.txt`, `/sitemap.xml`, неизвестный маршрут, варианты с trailing slash и query parameters, `/api/booking`.
- Статусы: `200` для страниц и системных файлов; `308` с `/privacy/` и `/consent/` на варианты без слеша; `404` для неизвестного URL; `405` для GET `/api/booking`.
- Исходники: `app/layout.tsx`, `app/page.tsx`, юридические страницы, `public/robots.txt`, `public/sitemap.xml`, `next.config.mjs`.
